Product Requirements Document · Phase 2 · Sprint 1

Build Blueprint:
The Living Execution Workspace

Converts the Builder agent's static 90-day roadmap from a wall of text into an interactive, trackable execution surface — checkboxes, progress state, milestone markers, and weekly check-ins. This is the single highest-leverage retention mechanic in the product. It is the reason a founder opens the app on Day 47 instead of never coming back after Day 1.

Phase 2 · Sprint 1 Target: 4 Weeks Stack: FastAPI · PostgreSQL · React (Antigravity) · Redis Depends On: Phase 1 Builder Agent Output Schema
North Star
Definition of Done

A founder who passed validation opens their roadmap and sees 90 individual, checkable tasks organized by week. They can mark a task complete, see their progress percentage update in real time, receive a prompt to check in weekly, and see the system flag when they've fallen behind schedule. The roadmap is no longer a document. It is a workspace the founder returns to daily. That daily return is Sprint 1 complete — nothing else in this sprint matters if that loop doesn't work.

Rule Zero

Do not rebuild the Builder agent's output logic. The 90-day roadmap generation already works. This sprint wraps a persistence and interaction layer around that existing output — it does not touch Agent 05's prompt or generation logic. If an engineer proposes changing how the roadmap is generated, that is scope creep. Reject it.

Sprint 1.1
S-1.1
Roadmap Persistence Layer
WEEK 1 · Engineer 2 leads

The Builder's 90-day JSON currently lives in Redis with a 24-hour TTL and disappears. Before anything else, that output needs to become permanent, queryable, per-task state in Postgres. This is the foundation every other piece of this sprint depends on.

  • Design and migrate the schema. Two new tables: roadmap_sessions (one row per validated idea) and roadmap_tasks (one row per day/task — 90 rows per session). See schema below.
  • Build the ingestion function. persist_roadmap(session_id, builder_output_json) — runs once, immediately after Agent 05 completes. Parses the 90-day array and inserts 90 rows into roadmap_tasks, each defaulting to status: pending.
  • Wire this into the existing pipeline. In the orchestrator, immediately after the Builder agent returns its output and before the SSE event fires to the frontend, call persist_roadmap(). This must complete before the "pipeline complete" event is sent — the frontend should load the workspace from Postgres, not from the raw Builder JSON.
  • Do not delete the Redis cache. Keep it as a fallback for the initial render while the Postgres write confirms. Belt and suspenders — this pipeline has already broken once on a rushed change.
// data_layer/schema_additions.sql
CREATE TABLE roadmap_sessions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), session_id UUID NOT NULL REFERENCES validation_sessions(id), user_id UUID, -- nullable until auth exists thesis_summary TEXT NOT NULL, total_days INT NOT NULL DEFAULT 90, started_at TIMESTAMPTZ DEFAULT NOW(), current_day INT DEFAULT 1, -- calendar day, not task index completion_pct NUMERIC(5,2) DEFAULT 0, last_checkin_at TIMESTAMPTZ, status TEXT DEFAULT 'active' -- active | paused | completed | abandoned ); CREATE TABLE roadmap_tasks ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), roadmap_session_id UUID NOT NULL REFERENCES roadmap_sessions(id), day_number INT NOT NULL, -- 1 to 90 week_number INT NOT NULL, -- 1 to 13 task_title TEXT NOT NULL, task_description TEXT, owner TEXT, -- from Builder output deliverable TEXT, status TEXT DEFAULT 'pending', -- pending | complete | skipped completed_at TIMESTAMPTZ, UNIQUE(roadmap_session_id, day_number) ); CREATE INDEX idx_roadmap_tasks_session ON roadmap_tasks(roadmap_session_id);
Sprint 1.1 Exit Gate

Running a test validation through the full pipeline results in exactly 90 rows in roadmap_tasks, correctly grouped by week, with the correct owner and deliverable fields populated from the Builder's JSON. Zero missing days. Zero duplicate day numbers.

Sprint 1.2
S-1.2
Task State API + Progress Calculator
WEEK 2 · Engineer 2 leads

This is the interaction layer. Everything a founder does inside the workspace — checking a box, viewing progress — routes through this API.

  • Build GET /workspace/{roadmap_session_id}. Returns all 90 tasks grouped by week, plus the parent session's completion percentage and current day. This is what the frontend loads on page open.
  • Build PATCH /workspace/tasks/{task_id}. Accepts {"status": "complete"} or {"status": "skipped"}. Updates the row, sets completed_at, and triggers a recalculation of the parent session's completion_pct.
  • Build the Progress Calculator service. A pure function: calculate_progress(roadmap_session_id) — counts complete/total tasks, computes percentage, and computes a slippage score: compare current_day (calendar days since started_at) against the day number of the last completed task. If the founder is on calendar Day 20 but has only completed tasks through Day 12, slippage = 8 days behind. This number drives the UI's slippage banner.
  • Write the daily cron job that advances current_day. A scheduled job (APScheduler or Cloud Scheduler) runs once daily, incrementing current_day for every active roadmap session. This must be a separate, deterministic process — do not calculate "current day" on the fly from timestamps in the API layer, or every timezone edge case becomes a support ticket.
# services/progress_calculator.py — core logic def calculate_progress(roadmap_session_id: str) -> dict: tasks = fetch_tasks(roadmap_session_id) total = len(tasks) completed = len([t for t in tasks if t.status == "complete"]) completion_pct = round((completed / total) * 100, 2) # Slippage: last completed day number vs calendar day elapsed last_completed_day = max( [t.day_number for t in tasks if t.status == "complete"], default=0 ) session = fetch_session(roadmap_session_id) calendar_days_elapsed = (now() - session.started_at).days + 1 slippage_days = calendar_days_elapsed - last_completed_day return { "completion_pct": completion_pct, "completed_tasks": completed, "total_tasks": total, "slippage_days": max(slippage_days, 0), "on_track": slippage_days <= 2 }
Sprint 1.2 Exit Gate

PATCH a task to complete, then GET the workspace — completion percentage updates correctly. Manually backdate a session's started_at in a test row to simulate being behind schedule; slippage_days reflects the correct gap.

Sprint 1.3
S-1.3
Weekly Check-in Scheduler
WEEK 3 · Engineer 2

The workspace only becomes a retention mechanic if the product reaches out — not the other way around. This is the piece that turns a passive dashboard into an active habit loop.

  • Build the weekly check-in cron job. Runs once per week per active session (calculate based on started_at, not calendar Monday — every founder's Week 1 starts on a different day). Pulls that founder's progress via the Progress Calculator.
  • Compose the check-in email via SendGrid. Content is dynamic: "You've completed 9 of 14 tasks this week. You're 2 days ahead of schedule" or "You've fallen 5 days behind. Here's what's still open." Include a direct link back to the workspace, pre-scrolled to the current week.
  • Log every check-in send to roadmap_sessions.last_checkin_at. Needed both for scheduling the next one and for a future analytics view on engagement.
  • Add an unsubscribe/pause option. A founder who wants to stop weekly emails should be able to without abandoning the roadmap itself. Add status: paused to stop check-ins but preserve all task data.
Sprint 1.3 Exit Gate

A test session artificially set to Week 2 triggers exactly one check-in email with the correct dynamic progress numbers. Running the cron job twice in the same week does not send a duplicate.

Sprint 1.4
S-1.4
Antigravity Workspace UI
WEEK 3–4 · Frontend Engineer

Four views. Build them in this order — each depends on the API from Sprints 1.1–1.2, not on each other.

  • View 1 — Weekly Kanban Board. Default landing view. 13 columns (weeks 1–13), each showing that week's tasks as checkable cards. Clicking a checkbox calls PATCH /workspace/tasks/{id} and optimistically updates the UI before the network call confirms.
  • View 2 — Progress Ring. A single circular progress indicator showing overall completion percentage, pinned to the top of the workspace. Updates in real time as tasks are checked.
  • View 3 — Slippage Banner. Conditionally rendered above the kanban board only when slippage_days > 2. Copy: "You're {slippage_days} days behind schedule. {N} tasks from earlier weeks are still open." Include a "Jump to oldest open task" button.
  • View 4 — Milestone Markers. At the end of weeks 4, 8, and 12, render a distinct milestone card summarizing what should be true by that point (per the Builder's original roadmap framing) — e.g., "By end of Week 4, you should have validated pricing with 10 real prospects." This anchors the day-by-day grind to larger checkpoints.
Sprint 1.4 Exit Gate

Full workspace renders correctly for a test session with a mix of complete, pending, and skipped tasks. Checking a box updates the progress ring within 1 second without a full page reload. Slippage banner correctly appears and disappears based on live data.

What Comes After This Sprint
Do Not Build Yet

Pivot history, shareable certificates, team workspaces, and the Execution Coach Agent all depend on this workspace existing first. Do not parallelize them. A founder cannot have a "pivot history" if there's no persistent execution state to compare against. Ship this sprint, confirm founders actually return on Day 7, then move to the next Phase 2 item.
