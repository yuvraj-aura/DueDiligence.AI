Phase 1 North Star
Definition of Done for Phase 1

A user inputs a raw idea. The five-agent pipeline runs sequentially without manual intervention. The Critic either halts with a cited kill reason or passes the idea to the Builder. The Builder outputs a structured 3-month roadmap. Every agent write triggers a telemetry log to the Data Layer. The Antigravity UI renders the final output. That full loop — working, in production, handling 10 concurrent validations — is Phase 1 complete. Nothing else counts.

Rule Zero for Every Engineer

The Data Layer telemetry write is not optional and not a Phase 2 task. Every agent that produces a signal — saturation score, viability ratio, kill reason — must write it anonymized to the database on the same run. If this slips to "later," the moat never compounds. Wire it in on week one, even if the schema is rough.

Sprint 1
S-01
Data Layer + FastAPI Gateway
WEEK 1–2 · Engineer 2 leads

Build the database and the API entry point first. Every subsequent agent depends on having a place to write signal data and a gateway to receive user input. This is not exciting work. It is the foundation everything else breaks without.

  • Design and migrate the PostgreSQL schema. Four tables minimum: validation_sessions, saturation_signals, cac_ltv_benchmarks, failure_taxonomy. Add session_hash (SHA-256 of session ID) as the anonymization key on every table from day one.
  • Build the async Telemetry Writer module. A single write_signal(table, payload) function that every agent calls. Non-blocking. Uses asyncpg. Fails silently with a structured error log — a DB write failure must never crash an agent run.
  • Build the FastAPI gateway. One endpoint: POST /validate. Accepts the five user inputs (thesis, niche, monetization, constraints, known competitors). Validates with Pydantic. Returns a session_id and kicks off the ADK pipeline asynchronously. Returns status via GET /status/{session_id}.
  • Wire up Redis for session state. The ADK orchestrator needs to persist agent outputs between pipeline steps. Store intermediate agent outputs keyed by session ID with a 24-hour TTL.
// data_layer/schema.sql — Minimum viable tables
CREATE TABLE validation_sessions ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), session_hash TEXT NOT NULL, -- anonymized: SHA256(session_id) niche_slug TEXT NOT NULL, -- normalized niche label pricing_model TEXT NOT NULL, -- usage | flat | freemium outcome TEXT, -- PASSED | KILLED | PENDING created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE saturation_signals ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), session_hash TEXT NOT NULL, niche_slug TEXT NOT NULL, saturation_score NUMERIC(4,2), -- 0.00–10.00 competitor_count INT, sentiment_delta NUMERIC(5,4), -- normalized -1 to +1 sources_cited INT, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE cac_ltv_benchmarks ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), session_hash TEXT NOT NULL, niche_slug TEXT NOT NULL, pricing_model TEXT NOT NULL, estimated_cac NUMERIC(10,2), estimated_ltv NUMERIC(10,2), viability_ratio NUMERIC(6,4), -- LTV/CAC created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE failure_taxonomy ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), session_hash TEXT NOT NULL, niche_slug TEXT NOT NULL, kill_reason TEXT NOT NULL, -- SATURATION | UNIT_ECON | NO_DIFFERENTIATOR | CITATION_FAIL saturation_score NUMERIC(4,2), retry_count INT DEFAULT 0, created_at TIMESTAMPTZ DEFAULT NOW() );
Sprint 1 Exit Gate

POST /validate accepts valid input and returns a session_id. A test write to all four DB tables succeeds. Redis stores and retrieves a mock session object. Do not start Sprint 2 until this gate clears.

Sprint 2
S-02
ADK Orchestrator + Agent 01: The Gatekeeper
WEEK 2–3 · Engineer 1 leads

Build the orchestration shell first, then slot in Agent 01. The ADK orchestrator is the conductor. Build it so that adding the remaining four agents is a mechanical task, not an architectural decision.

  • Initialize the ADK SessionService and Runner. Use InMemorySessionService for development. Switch to DatabaseSessionService backed by PostgreSQL before production. The Runner manages the sequential pipeline handoffs.
  • Define the canonical inter-agent message schema. Every agent receives a PipelineContext object and must return an updated PipelineContext. Define this Pydantic model now — it is the contract every agent signs. Fields: session_id, raw_input, gatekeeper_output, researcher_output, numbers_output, critic_decision, builder_output, pipeline_status.
  • Build Agent 01 — The Gatekeeper. Gemini prompt: takes the five raw inputs and outputs a strict JSON object: normalized niche slug, locked value proposition (≤40 words), primary audience definition, and a list of three falsifiable assumptions the validation must test. Strict JSON output only — no prose.
  • Wire Agent 01 into the orchestrator. On POST /validate, the orchestrator invokes Agent 01, stores the output in Redis under {session_id}:gatekeeper, and triggers Agent 02. The FastAPI /status endpoint reads Redis and returns current pipeline stage.
# agents/gatekeeper.py — Core structure from google.adk.agents import LlmAgent from google.adk.models import Gemini GATEKEEPER_PROMPT = """ You are a precision input normalizer for a market validation system. Given raw user input, output ONLY a valid JSON object with these exact fields: { "niche_slug": "kebab-case-niche-label-max-5-words", "value_proposition": "One sentence, max 40 words, subject-verb-object format", "target_audience": "Specific descriptor. No vague terms like 'businesses' or 'users'.", "pricing_model": "usage_based | flat_subscription | freemium", "falsifiable_assumptions": [ "Assumption 1 that market research must confirm or deny", "Assumption 2", "Assumption 3" ], "runway_weeks": integer, "marketing_budget_usd": integer } Return ONLY the JSON. No preamble. No explanation. No markdown. """ gatekeeper_agent = LlmAgent( name="gatekeeper", model=Gemini("gemini-2.0-flash"), instruction=GATEKEEPER_PROMPT, output_key="gatekeeper_output" )
Sprint 2 Exit Gate

A raw user input posted to /validate triggers Agent 01, which returns a valid structured JSON object in under 8 seconds. The output is stored in Redis. /status/{session_id} returns "stage": "gatekeeper_complete".

Sprint 3
S-03
MCP Servers + Agent 02: The Researcher
WEEK 3–5 · Engineer 1 leads · Engineer 2 supports MCP

This is the hardest sprint. The MCP servers are custom-built and directly determine the quality of every validation. The Researcher is the only agent that touches the real world. Get this wrong and the entire system produces confident garbage.

  • Build MCP Server 1: Web Competitor Scraper. Uses SerpAPI to search for "{niche_slug} software", "{niche_slug} tool", "best {niche_slug} alternatives". Returns: list of competitor names, URLs, estimated founding year, and G2/Capterra review count if available. Cap at 20 results. Structured JSON output only.
  • Build MCP Server 2: Reddit Sentiment Scraper. Queries Reddit API for the top 50 posts in the last 12 months matching the niche slug across r/entrepreneur, r/SaaS, r/startups, r/indiehackers. Extracts comment sentiment (positive/negative/neutral via Gemini Flash). Returns a normalized sentiment delta: +1.0 (all positive) to -1.0 (all negative).
  • Build MCP Server 3: Product Hunt + GitHub Signal. Product Hunt API: count products launched in this niche in the last 18 months. GitHub: count public repos matching niche keywords created in last 12 months. Both are proxy signals for developer-perceived market opportunity.
  • Build Agent 02 — The Researcher. Calls all three MCP servers in parallel (use asyncio.gather). Computes the Saturation Score using the weighted formula below. Every output must include the source URL that justifies it. Unsourced claims = automatic Critic kill.
  • Wire telemetry write after Agent 02 completes. Call write_signal("saturation_signals", payload) with the saturation score, competitor count, and sentiment delta. This is the first Data Layer write in the pipeline.
# Saturation Score Formula (0–10) # Engineer 1 hardcodes this as a deterministic function — not AI-generated def compute_saturation_score( competitor_count: int, # from SerpAPI ph_launches_18mo: int, # from Product Hunt gh_repos_12mo: int, # from GitHub sentiment_delta: float # from Reddit: -1.0 to +1.0 ) -> float: # Competitor density: 0–4 points comp_score = min(competitor_count / 10, 1.0) * 4.0 # Recent market activity: 0–3 points activity_score = min((ph_launches_18mo + gh_repos_12mo) / 30, 1.0) * 3.0 # Sentiment: negative sentiment = more saturated / frustrated market = higher score # Positive sentiment = underserved, eager market = lower score sentiment_score = (1.0 - ((sentiment_delta + 1.0) / 2.0)) * 3.0 return round(comp_score + activity_score + sentiment_score, 2)
Sprint 3 Exit Gate

For a test niche (e.g. "invoicing software for freelance designers"), Agent 02 returns: a competitor list with at least 5 sourced entries, a sentiment delta with cited Reddit posts, and a Saturation Score between 0–10. Total agent runtime under 30 seconds. Telemetry write to saturation_signals confirms in DB.

Sprint 4
S-04
Agent 03: The Numbers Engine
WEEK 5–6 · Engineer 1

The Numbers Engine does not browse the web. It reasons over the Researcher's output and the user's stated constraints. Its output is a deterministic financial model, not an AI opinion.

  • Define CAC estimation logic. Based on marketing budget and pricing model, estimate CAC using a lookup table of industry benchmarks by pricing tier (not AI-hallucinated — hardcode from known SaaS benchmarks: usage-based CAC avg $180, flat sub avg $240, freemium avg $310). The AI adjusts these by the niche sentiment delta — positive sentiment = lower CAC multiplier.
  • Define LTV estimation logic. Based on stated pricing model and niche, estimate average MRR per customer and apply a churn rate lookup by pricing model (freemium: 8%/mo, flat: 4%/mo, usage: 5%/mo). LTV = (MRR per user) / churn rate.
  • Compute the Viability Ratio. viability_ratio = LTV / CAC. Ratios below 3.0 are flagged as structurally broken. Ratios between 3.0–5.0 are marginal. Above 5.0 is viable. This output goes directly to the Critic as a binary signal.
  • Wire telemetry write. Write estimated_cac, estimated_ltv, and viability_ratio to cac_ltv_benchmarks. This is the Data Layer's most commercially valuable signal — it is how the future benchmark product gets built.
Sprint 4 Exit Gate

For a test case with a $500 marketing budget and flat-rate pricing, Agent 03 returns a viability ratio with explicit CAC and LTV figures and a VIABLE / MARGINAL / BROKEN classification. Telemetry write confirms in cac_ltv_benchmarks.

Sprint 5
S-05
Agent 04: The Failsafe Critic
WEEK 6–7 · Engineer 1

The Critic is the most important agent and the simplest to build. It applies rules, not intelligence. Two hard kill conditions. No exceptions. No "on the other hand." Build it as a rule engine first, then layer Gemini for the kill reason narrative.

  • Kill Condition 1 — Citation Failure. Check the Researcher output: count the number of competitor entries that have a valid source URL. If more than 20% of claims are unsourced, return KILL: CITATION_FAIL. Do not pass to Agent 05.
  • Kill Condition 2 — Saturation Threshold. If saturation_score > 8.0 AND no unique differentiator was identified by the Gatekeeper, return KILL: SATURATION_NO_DIFFERENTIATOR. Do not pass to Agent 05.
  • Kill Condition 3 — Unit Economics. If viability_ratio < 3.0, return KILL: UNIT_ECON_BROKEN. This is not a soft warning. It is a hard stop.
  • On PASS: Gemini generates a one-paragraph "Thesis Confirmation" that summarizes why the idea cleared all three gates — with citations. This is the user's first positive signal after the pipeline runs.
  • Telemetry write on KILL only. Write kill reason, saturation score, and session hash to failure_taxonomy. This is the most analytically valuable signal in the entire system — it builds the Failure Pattern Index that VCs will eventually pay for.
Sprint 5 Exit Gate

Run three test cases: one that should kill on saturation, one on unit economics, one that should pass. All three produce the correct outcome. Kill cases write to failure_taxonomy. Pass case generates a Thesis Confirmation paragraph with citations.

Sprint 6
S-06
Agent 05: The Builder + Antigravity UI
WEEK 7–9 · Engineer 1 (Builder) + Frontend Engineer (Antigravity)

The Builder's output is a structured JSON object. Antigravity consumes that JSON and renders it. These are two separate concerns — build and test them independently.

  • Define the Builder output schema. The JSON must contain: executive_summary (3 sentences), roadmap (array of 90 objects, one per day, each with day, week, task, owner, deliverable), risk_matrix (top 5 risks with probability and impact scores), financial_model (CAC, LTV, break-even month, MRR projections at 3/6/12 months), cited_sources (full list from Researcher).
  • Prompt the Builder agent. Gemini takes the full PipelineContext (all prior agent outputs) and generates the roadmap. Instruct it to populate every single day — no "continue this pattern" shortcuts. Verify day count in post-processing before storing.
  • Build the Antigravity UI components. Four views: (1) Pipeline status tracker — shows which of the 5 agents is running in real time. (2) Kill screen — rendered when the Critic fires, showing the exact kill reason and the cited evidence. (3) Roadmap view — interactive 90-day calendar. (4) Dashboard — risk matrix, financial model, and source index.
  • Wire the Antigravity trigger. When Agent 05 writes its output to Redis, the FastAPI gateway emits a Server-Sent Event (SSE) to the Antigravity frontend. Antigravity listens on the SSE stream and triggers the render when the event arrives.
Sprint 6 Exit Gate

A passing idea produces a complete 90-day roadmap JSON with no missing days. Antigravity renders all four UI views without errors. A killed idea renders the Kill screen with cited evidence. End-to-end pipeline runtime: under 90 seconds.

Sprint 7
S-07
Hardening, Load Testing & Production Deploy
WEEK 9–12 · Both Engineers

The pipeline works in a single thread. Now make it not break under 10 concurrent users. This sprint is not about new features. It is about not embarrassing yourself in the first week of beta.

  • Add global error handling and retry logic. Every MCP server call gets 3 retries with exponential backoff. Every Gemini API call gets 2 retries. A failed agent writes its error state to Redis and exposes it via /status. The pipeline never silently hangs.
  • Load test with Locust. Simulate 10 concurrent validation sessions. Target: all 10 complete within 3 minutes. Identify and fix the bottleneck — it will almost certainly be the Researcher's MCP server calls hitting external API rate limits.
  • Implement API rate limit handling for all external services. Reddit: 60 requests/minute. Product Hunt: 60 requests/hour (cache aggressively). SerpAPI: per your plan tier. GitHub: 5000 requests/hour (authenticated). Add per-service rate limit tracking in Redis.
  • Deploy to Google Cloud Run. Two services: api-service (FastAPI) and agent-worker (ADK pipeline). Cloud Run auto-scales. Set minimum instances to 1 to avoid cold starts on the agent worker. Antigravity deploys separately per its own deployment model.
  • Set up basic monitoring. Cloud Monitoring for API latency and error rates. Postgres slow query log for any Data Layer write taking over 200ms. Alert on pipeline failure rate above 5% (excluding intentional Critic kills).
Phase 1 Final Exit Gate

10 concurrent validations complete without errors. All four DB tables are accumulating signal data. The Antigravity UI renders correctly on completion. Pipeline failure rate (excluding Critic kills) is under 2%. This is Phase 1 complete.
